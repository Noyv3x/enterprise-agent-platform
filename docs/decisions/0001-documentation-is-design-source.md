# 0001：文档是唯一设计真相源

- 状态：accepted
- 日期：2026-07-21

## 背景

过去，仓库规则分散在根 README、组件 README、`AGENTS.md`、`CLAUDE.md`、安全审计和代码注释中。相同设计被重复描述，配置、模型、超时和部署行为容易在实现变化后留下旧版本。`CLAUDE.md` 又只是另一个规则文件的跳转，不能形成可靠的产品知识层。

代码只能说明当前实现，不能独立表达产品意图、被拒绝的替代方案和跨组件不变量。另一方面，只声明“先写文档”也无法阻止文档和实现长期漂移。

## 决定

`docs/` 成为 ubitech agent 的唯一设计真相源：

1. `design/` 定义产品和架构不变量；
2. `reference/` 定义当前配置、协议和数据布局；
3. `operations/` 定义部署与更新行为；
4. `development/` 定义实现和验证流程；
5. `contracts/` 保存需要由代码和测试共同消费的精确机器策略；
6. `decisions/` 记录重要选择的理由；
7. `audits/` 只保存带日期的历史检查，不冒充当前规范。

设计变更顺序为：先修改规范或契约，再修改代码，最后更新测试、生成资源和操作说明。一个可合并变更必须包含所有同步结果。

代码路径与文档 domain 通过机器可读映射关联。CI 检查相对链接、映射覆盖和代码/文档成对变化。Run 空闲、模型轮次和 terminal 默认超时这三项跨层策略只在 `contracts/runtime-policy.json` 定义，叙述文档通过链接引用；其它策略由各自设计、参考与测试共同约束。

根 README 和组件 README 只保留项目入口、最短启动命令和 canonical docs 链接。`AGENTS.md` 与 `CLAUDE.md` 删除并弃用，不再承载仓库规则。

## 后果

优点：

- 产品意图、运行契约与实现边界有统一入口；
- 配置和时间策略可以机器校验，减少复制漂移；
- 代码审核可以先判断设计变化，再判断实现是否完整；
- 新成员不需要从多个 Agent 专用规则文件拼接上下文。

成本：

- 功能变更必须同时维护文档和映射；
- 文档组织本身需要 review；
- 外部编码工具不一定自动读取 `docs/`，根 README 必须提供醒目入口；
- CI 能检查最终 diff 是否成对，不能证明作者真实的编辑时间顺序，顺序仍依赖开发流程和评审。

## 例外

历史审计、文档索引与 ADR 本身不直接映射生产代码，可以独立维护。已登记为代码域真相源的规范和受管生产代码必须通过双向共改检查，不提供 docs-only/code-only 绕过；纯排版修正也应随该领域下一次同步变更一起交付，避免例外逐步掏空门禁。

紧急安全修复允许先阻断漏洞，但同一修复交付中仍必须补齐规范和测试；不能以紧急为由永久让实现成为第二套真相源。
