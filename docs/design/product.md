# 产品设计

本文定义 ubitech agent 当前产品边界。系统结构见[系统架构](system-architecture.md)，Agent 行为见 [Agent Runtime](agent-runtime.md)，安全前提见[安全与信任边界](security-and-trust.md)。

## 产品目标

ubitech agent 面向彼此可信的小规模成员，为每位用户提供私人 Agent，并在共享频道中提供频道 Agent。系统将对话、文件、工作环境、会话、记忆、技能和浏览器状态按主 Agent 分开，同时允许 Agent 为单次操作显式使用宿主机能力完成真实工作。

每个私人 Agent 和频道主 Agent都有独立持久工作目录与 Sandbox 容器，委派子 Agent共享父环境。Sandbox 用于环境隔离和减少误操作，不是恶意多租户安全边界；显式宿主执行可以获得部署用户及其免密 sudo 权限，接受这一信任风险是部署前提。

## 角色与权限

平台内置管理员、经理、成员和只读四组权限组合。服务端权限判断是唯一授权边界，前端隐藏入口只用于改善体验。

- 管理员负责账号、模型授权、运行时、自动更新、Telegram、审计和系统设置。
- 经理可管理频道和知识库，并使用私人及频道 Agent。
- 成员可参与频道并使用私人 Agent。
- 只读用户只能读取被授权的频道和知识内容。

账号还可以分别保存显示名称、职位、模型选择、思考深度、时区和启用状态。Agent 提示词必须携带当前说话人的可用上下文，但面向用户始终以“ubitech agent”自称，不暴露内部框架、Runtime 或模型供应商实现。

## 对话模型

私人 Agent 和每个频道主 Agent 都是长期逻辑身份，拥有稳定工作区和可轮换的会话生命周期。私人消息按进入顺序持久化；同一用户连续发送的相关短消息可并入正在运行的回复，并保留每条消息的处理状态。频道消息、私人消息和 Agent 回复都保存在平台消息库中。

界面只在发生工具调用时展示工作记录。工作过程先展开显示在当前回复顶部，最终输出开始后自动折叠；没有工具调用时不创建空工作记录。审批请求是工作过程的一部分，但必须保持为明确、可操作的独立状态。

## 核心能力

- 频道、私人对话、附件、输入合并、流式回复和上下文用量查看。
- 账号、权限组、频道、私人消息和 token 用量管理。
- 本地知识库，以及可选的 Cognee 增强。
- Agent 记忆、用户资料记忆、候选记忆审核和跨会话搜索。
- 每 Agent 技能包，包括只读预置技能和用户技能。
- 私人 Agent 的一次性、间隔和 cron 计划任务。
- 基于 Camoufox 的浏览器工具与低帧率只读预览。
- 运行中终端的只读预览。
- 每主 Agent 独立、可停止后重建且保留工作目录/HOME/环境的 Sandbox。
- 仅面向私人聊天的 Telegram Gateway。
- Codex OAuth 与 Grok OAuth；不提供模型 API key 产品入口。
- 等待空闲后执行、全程阻断使用并可失败回滚的自动更新。

详细集成语义见[外部集成](integrations.md)，数据所有权见[数据、记忆与会话](data-memory-sessions.md)。

## 界面范围

浏览器界面支持简体中文、English 和繁體中文。i18n 只约束界面文字，不翻译用户消息、Agent 输出、文件内容、工具输出或日志。响应式布局必须使用完整可用视口，在手机、平板和桌面上都不能依赖固定桌面宽度。

浏览器与终端预览属于聊天界面的侧栏内容。没有打开的浏览器 tab 或运行中的终端时，不显示相应侧栏按钮；资源存在但首帧仍在加载时，应显示加载态，而不是误报“尚未运行”。前端细则见[前端设计](frontend.md)。

## 明确不做

- 不把 Agent Sandbox 宣称为恶意多租户安全边界。
- 不将 Cognee、Firecrawl 或 Pi 作为可直接修改的产品代码。
- 不让浏览器或终端预览成为可交互的远程控制入口。
- 不让 OAuth 在线发现绕过 Runtime 的模型能力目录。
- 不在界面之外扩展本轮 i18n 范围。
